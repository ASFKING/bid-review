## 一、系统定位

### 1.1 一句话定义

基于大模型的标书智能审核系统，自动解析 Word 标书文档，通过多维度 AI 审核找出合规性、完整性、报价、风险等问题，输出审核报告并直接在 Word 文档中标注问题位置。

### 1.2 核心价值

| 角色 | 痛点 | 解决方案 |
|------|------|---------|
| 审核人员 | 标书动辄几百页，人工逐页审核耗时耗力 | AI 自动审核，人只做判断和决策 |
| 项目经理 | 审核标准不统一，容易遗漏 | 结构化审核维度，标准化输出 |
| 管理层 | 无法量化审核质量 | 评分体系 + 问题统计 |

### 1.3 学习目标

```
阶段目标：
├── 阶段 1：会用 LlamaIndex 处理文档（加载、切分、索引）
├── 阶段 2：会用 LLM API 做结构化输出（Prompt 工程）
├── 阶段 3：会编排多步骤 AI 工作流（Agent 编排）
├── 阶段 4：会做人工干预和结果校验（人机协同）
└── 阶段 5：会做输出渲染（报告生成、文档标注）
```

### 1.4 设计理念：Claude Code 概念映射

本系统的设计理念借鉴 Claude Code 的核心概念，将其映射到多智能体审核场景：

| Claude Code 概念 | 本系统映射 | 解决的问题 |
|---|---|---|
| **CLAUDE.md** | `system_config.yaml` 系统级配置 | 全局规范统一，所有 Agent 共享同一份"入职文档" |
| **Context Window 管理** | 两级审核策略 | 避免长文档摘要丢失关键细节 |
| **Sub Agents** | 各维度审核 Agent | 独立上下文窗口，互不污染 |
| **Agent Teams** | ReviewTeam 去中心化协作 | Agent 间可直接通信，不只汇报给主 Agent |
| **Skills** | Prompt 模板库 | 可复用的专家级指令，而非一次性提示词 |
| **Hooks** | 事件驱动规则校验 | 纯规则检查零 token 成本，毫秒级响应 |
| **Extended Thinking** | Agent 推理预算 | 复杂问题先想后做，避免盲目输出 |
| **Checkpoints** | 审核流程快照 | 支持回滚、断点续审、错误恢复 |
| **Worktrees** | 并行审核隔离 | 多 Agent 并行工作互不干扰 |

### 2.2 为什么选 LlamaIndex

| 能力 | LlamaIndex 提供 | 自己写的工作量 |
|------|----------------|---------------|
| Word/PDF 文档加载 | ✅ DocxReader / PDFReader | 省 200 行 |
| 长文档切分 | ✅ SentenceSplitter / TokenTextSplitter | 省 150 行 |
| LLM 统一调用 | ✅ OpenAI / Qwen / DeepSeek 适配 | 省 300 行 |
| 结构化输出 | ✅ PydanticOutputParser | 省 200 行 |
| 向量索引 | ✅ VectorStoreIndex | 省 500 行 |
| 摘要提取 | ✅ SummaryIndex | 省 200 行 |

**总计：LlamaIndex 帮你省了约 1500 行基础设施代码，让你专注于业务逻辑。**

## 四、核心设计：system_config.yaml（类比 CLAUDE.md）

### 4.1 设计理念

Claude Code 中 CLAUDE.md 的核心价值：**写一次，终身受益**。每个 Agent 启动时自动读取这份"入职文档"，保证行为一致性。

本系统的 `system_config.yaml` 承担同样角色——全局规范、模型配置、输出格式、安全策略，统一管理。

### 4.2 配置结构

```yaml
# bid-review/system_config.yaml

# ============ 系统身份 ============
system:
  name: "标书智能审核系统"
  version: "2.0"
  description: "基于大模型的标书多维度智能审核平台"

# ============ 全局审核规范 ============
# 所有 Agent 共享的行为准则，类似 CLAUDE.md 中的项目规范
global_rules:
  - "只报告有明确证据的问题，禁止猜测"
  - "每个 issue 必须引用原文证据"
  - "confidence < 0.7 的问题标记为 needs_verification"
  - "禁止重复报告同一问题"
  - "严重程度必须基于实际影响判断，不夸大不缩小"
  - "资质/门槛问题默认高严重度"
  - "格式问题默认低严重度"

# ============ 输出格式约束 ============
output_schema:
  type: "json"
  required_fields:
    - id
    - dimension
    - severity
    - title
    - description
    - evidence
    - location
    - suggestion
    - confidence
  severity_enum: ["高", "中", "低"]
  risk_level_enum: ["高", "中", "低"]
  confidence_range: [0.0, 1.0]

# ============ 模型配置 ============
# 分层使用，类似 Claude Code 的 model 选择策略
models:
  reasoning:
    name: "qwen-max"
    use_for: ["行业分析", "审核判断", "结果校验"]
    description: "高质量推理，用于复杂判断"
  fast:
    name: "qwen-turbo"
    use_for: ["章节摘要", "报告生成"]
    description: "快速处理，用于信息压缩和格式化"

# ============ 审核维度配置 ============
dimensions:
  completeness:
    name: "完整性"
    enabled: true
    priority: "高"
    description: "检查投标文件是否完整响应招标要求"
    section_keywords: ["技术要求", "商务要求", "投标响应"]
  compliance:
    name: "合规性"
    enabled: true
    priority: "高"
    description: "检查资质、证书、人员是否满足要求"
    section_keywords: ["资格审查", "资质证明", "人员配置"]
  price:
    name: "报价"
    enabled: true
    priority: "中"
    description: "分析报价合理性和完整性"
    section_keywords: ["报价清单", "费用明细", "分项报价"]
  risk:
    name: "风险"
    enabled: true
    priority: "中"
    description: "识别合同条款中的潜在风险"
    section_keywords: ["合同条款", "违约责任", "付款条件"]

# ============ Hooks 规则配置 ============
hooks:
  enabled: true
  rules:
    id_uniqueness:
      description: "问题 ID 唯一性检查"
      action: "auto_fix"
    evidence_length:
      description: "evidence 字段长度检查"
      min_length: 10
      action: "flag_for_review"
    severity_confidence_check:
      description: "高严重度 + 低置信度告警"
      severity_threshold: "高"
      confidence_threshold: 0.8
      action: "warn"
    score_consistency:
      description: "评分与问题数一致性检查"
      high_issue_threshold: 5
      score_threshold: 70
      action: "warn"
    required_fields:
      description: "必填字段检查"
      action: "flag_for_review"

# ============ 敏感文件拒绝列表 ============
# 类比 Claude Code 的 deny list
deny_access:
  - "*.env"
  - "secrets/**"
  - "credentials.json"
  - "*.key"

# ============ 评分配置（v2.1 新增）============
scoring:
  base_score: 100
  severity_deduction:
    高: 15
    中: 5
    低: 1
  dimension_weights:
    完整性: 0.35
    合规性: 0.30
    报价: 0.20
    风险: 0.15
  confidence_factor:
    ">=0.9": 1.0
    "0.8-0.9": 0.8
    "0.7-0.8": 0.6
    "<0.7": 0.3
  risk_thresholds:
    低风险: 85
    中风险: 60
    # < 60 为高风险

# ============ LLM 容错配置（v2.1 新增）============
llm_resilience:
  max_retries: 2
  retry_delays: [2, 4]              # 指数退避秒数
  timeout_seconds: 60
  rate_limit_max_wait: 30           # 限流最大等待秒数
  json_repair: true                 # 启用 JSON 自动修复
  fallback_strategy: "default_recipe"  # Analyzer 失败时的降级配方
```

## 五、数据结构定义

### 5.1 文档解析层数据结构

```python
from dataclasses import dataclass, field

@dataclass
class TableData:
    """表格数据"""
    headers: list[str]                    # 表头
    rows: list[list[str]]                 # 数据行
    as_text: str                          # 文本描述版（给 LLM 看）
    location: str                         # 所在章节

@dataclass
class Section:
    """文档章节"""
    title: str                            # 章节标题
    level: int                            # 标题层级 1/2/3
    content: str                          # 正文文本
    tables: list[TableData]               # 该章节内的表格
    page_range: tuple[int, int]           # 起止页码
    children: list['Section']             # 子章节

@dataclass
class ParsedDocument:
    """解析后的文档"""
    filename: str
    total_pages: int
    sections: list[Section]
    full_text: str                        # 全文纯文本
    metadata: dict                        # 文件元信息
```

### 5.2 智能层数据结构

```python
from enum import Enum

class Severity(Enum):
    HIGH = "高"
    MEDIUM = "中"
    LOW = "低"

class RiskLevel(Enum):
    HIGH = "高"
    MEDIUM = "中"
    LOW = "低"

@dataclass
class DimensionConfig:
    """审核维度配置"""
    name: str                             # 维度名
    enabled: bool                         # 是否启用
    priority: str                         # 优先级
    focus_areas: list[str]                # 关注点列表
    section_keywords: list[str]           # 关联章节关键词

@dataclass
class AgentDefinition:
    """Agent 定义"""
    agent_id: str
    agent_name: str
    dimension: str
    description: str
    skill_id: str                         # 对应的 Skill 模板 ID

@dataclass
class ReviewRecipe:
    """审核配方（Analyzer 输出）"""
    industry: str                         # 行业类型
    sub_industry: str                     # 细分行业
    confidence: float                     # 置信度
    dimensions: list[DimensionConfig]     # 审核维度
    agent_definitions: list[AgentDefinition]  # Agent 定义

@dataclass
class Issue:
    """单个问题"""
    id: str                               # 问题编号
    dimension: str                        # 审核维度
    severity: Severity                    # 严重程度
    title: str                            # 问题标题
    description: str                      # 问题描述
    evidence: str                         # 原文证据
    location: str                         # 问题位置
    suggestion: str                       # 修改建议
    confidence: float                     # 确信度 0-1
    # v2.0 新增字段
    discovered_by: str                    # 发现者 Agent
    cross_verified_by: list[str] = field(default_factory=list)  # 交叉验证者
    needs_manual_review: bool = False     # 是否需要人工确认
    verification_note: str = ""           # 校验备注
    hook_flags: list[str] = field(default_factory=list)  # Hook 标记

@dataclass
class ReviewResult:
    """单个 Agent 的审核结果"""
    agent_name: str
    dimension: str
    issues: list[Issue]
    summary: str
    score: float                          # 0-100
    risk_level: RiskLevel
    needs_verification: list[str]         # 需回查的 issue id
    # v2.0 新增
    warnings: list[str] = field(default_factory=list)  # Hook 告警

@dataclass
class ValidatedResult:
    """校验后的结果"""
    original: ReviewResult
    confirmed_issues: list[Issue]
    rejected_issues: list[Issue]          # 幻觉
    validation_notes: str

@dataclass
class AuditReport:
    """最终审核报告"""
    document_name: str
    overall_score: float
    overall_risk: RiskLevel
    dimension_scores: dict[str, float]
    all_issues: list[Issue]
    summary: str
    # v2.0 新增
    hook_summary: dict                    # Hook 执行统计
    checkpoint_history: list[dict]        # 检查点历史
```

    state: dict                           # 完整状态快照

@dataclass
class LLMCallLog:

## 六、核心模块设计

### 6.1 Hooks 机制（事件驱动规则校验）

#### 6.1.1 设计理念

Hooks 的核心思想：**把确定性检查从 LLM 流程中剥离出来。**

- **Hooks** = 潜水器的自动报警系统 → 快、免费、确定性
- **LLM 调用** = 打电话给海洋生物学家 → 慢、有成本、有随机性

能用规则解决的，绝不动用 AI。AI 只处理规则搞不定的事。

#### 6.1.2 Hook 类型清单

| Hook 名称 | 触发时机 | 检查内容 | 动作 | 是否消耗 Token |
|-----------|---------|---------|------|--------------|
| `id_uniqueness` | issue 创建时 | ID 是否重复 | 自动修正 | ❌ |
| `evidence_length` | issue 创建时 | evidence 是否过短 | 标记待确认 | ❌ |
| `severity_confidence` | issue 创建时 | 高严重度+低置信度 | 告警 | ❌ |
| `field_validation` | issue 创建时 | 必填字段是否缺失 | 标记待确认 | ❌ |
| `enum_check` | issue 创建时 | 枚举值是否合法 | 自动修正 | ❌ |
| `score_consistency` | result 生成时 | 评分与问题数是否矛盾 | 告警 | ❌ |
| `duplicate_detection` | result 生成时 | 同一位置是否重复报告 | 合并去重 | ❌ |
| `issue_count_check` | result 生成时 | 问题数量是否异常 | 告警 | ❌ |
| `cross_reference` | issue 入共享上下文时 | 是否与其他 Agent 发现的问题重复 | 提升置信度 | ❌ |

#### 6.1.3 Hook 执行流程

```
Agent 产出 issue
       │
       ▼
  ┌─────────────────────────────────────────────┐
  │  Hooks 引擎（纯规则，零 token，毫秒级）       │
  │                                             │
  │  ① ID 唯一性检查                             │
  │  ② evidence 长度检查                         │
  │  ③ 严重度-置信度一致性检查                    │
  │  ④ 必填字段完整性检查                         │
  │  ⑤ 枚举值合法性检查                           │
  │  ⑥ 重复检测（与共享上下文比对）               │
  │                                             │
  │  输出：                                       │
  │  ├── 能修正的 → 自动修正（ID 重复、格式错误）  │
  │  ├── 能告警的 → 标记警告（评分矛盾）          │
  │  └── 不能判断的 → 标记待 LLM 处理             │
  └──────────────────┬──────────────────────────┘
                     │
                     ▼
              ┌─────────────┐
              │  LLM 校验    │  ← 只处理 Hook 搞不定的事
              │ (按需触发)    │
              └─────────────┘
```

#### 6.1.4 Hook 与 Validator 的关系

```
原来的设计（v1.0）：
  所有 Agent 完成 → Validator 统一校验（全量 LLM 调用）

改进后的设计（v2.0）：
  每个 Agent 产出 issue → Hooks 实时拦截（零成本）→ 低置信度问题才交给 LLM
```

Hooks 承担了 Validator 80% 的工作量（纯规则检查），Validator 只处理剩下 20% 需要语义理解的问题。

### 6.3 Context Window 管理（两级审核策略）

#### 6.3.1 设计理念

Claude Code 的核心教训：**上下文过长会导致 "Context Rot"，输出质量下降。**

v1.0 的"300 字摘要 → 审核"策略就是典型的 Context Rot 诱因。v2.0 改为两级审核：

- **第一轮（粗筛）**：基于摘要快速定位可疑区域
- **第二轮（精审）**：对可疑区域的原文做精细审核（不做摘要）

类比深海捕鱼：先用声呐扫描大范围鱼群，再精确下网。

#### 6.3.2 两级审核流程

```
原始章节（15000 字）
    │
    ▼
判断：内容长度 ≤ 8000 字？
    ├── 是 → 直接用原文审核（避免信息损失）
    └── 否 → 进入两级审核
              │
              ▼
    ┌─────────────────────────────────┐
    │  第一轮：粗筛                    │
    │                                 │
    │  LlamaIndex SentenceSplitter    │
    │  切分为多个 chunk（≤8000 字）    │
    │  相邻 chunk 重叠 500 字          │
    │         │                       │
    │         ▼                       │
    │  每个 chunk 生成摘要             │
    │  （不是 300 字随便压，            │
    │   而是按维度提取关键信息）        │
    │         │                       │
    │         ▼                       │
    │  基于摘要快速扫描                │
    │  标记可疑 chunk                  │
    │  策略：宁漏勿误                  │
    └─────────────┬───────────────────┘
                  │
                  ▼
    ┌─────────────────────────────────┐
    │  第二轮：精审                    │
    │                                 │
    │  只对可疑 chunk 的原文做审核      │
    │  不做摘要，直接审原文             │
    │  要求精确引用原文证据             │
    │                                 │
    │  输出：Issue 列表                │
    └─────────────┬───────────────────┘
                  │
                  ▼
    ┌─────────────────────────────────┐
    │  向量回查验证                    │
    │                                 │
    │  对低置信度问题（<0.7）          │
    │  用 LlamaIndex 向量索引          │
    │  回查原文验证证据是否存在         │
    └─────────────────────────────────┘
```

#### 6.3.3 上下文压缩策略

类比 Claude Code 的 `/compact` 命令：

| 场景 | 策略 | 类比 |
|------|------|------|
| 单章节过长 | 两级审核（粗筛+精审） | Context Window 管理 |
| Agent 上下文膨胀 | 压缩历史，保留关键发现 | `/compact` |
| 跨 Agent 共享信息 | 只共享 issue 列表，不共享原始对话 | 共享上下文精简 |

### 6.5 Checkpoints 机制

#### 6.5.1 设计理念

类比 Claude Code 的 Checkpoints：每个关键步骤自动保存快照，支持回滚和断点续审。

#### 6.5.2 检查点清单

| 检查点 | 时机 | 保存内容 | 回滚场景 |
|--------|------|---------|---------|
| `doc_parsed` | 文档解析完成 | ParsedDocument | 解析结果有问题，重新解析 |
| `recipe_generated` | Analyzer 完成 | ReviewRecipe | 行业判断错误，重新分析 |
| `recipe_confirmed` | 用户确认配方 | ReviewRecipe（已确认） | 用户修改了配方 |
| `agents_created` | Agent 团队创建完成 | Agent 列表 | 需要增删 Agent |
| `review_completed` | 所有 Agent 审核完成 | ReviewResult 列表 | 审核结果有误 |
| `validation_completed` | Validator 校验完成 | ValidatedResult 列表 | 校验过度/不足 |
| `report_generated` | 报告生成完成 | AuditReport | 需要重新生成报告 |

#### 6.5.3 回滚与断点续审

```
回滚流程：
  用户选择回滚到检查点 N
    → 删除检查点 N 之后的所有数据
    → 恢复检查点 N 的状态
    → 从检查点 N 继续执行

断点续审：
  系统异常中断
    → 读取最近的检查点
    → 恢复状态
    → 从中断处继续执行
```

### 6.7 LLM 调用容错策略

#### 6.7.1 设计理念

LLM 是整个系统中最不可靠的环节——网络超时、返回格式错误、限流、幻觉，任何一种都可能发生。容错设计的核心原则：**永远不要假设 LLM 会正常返回。**

类比深海探测：你可以规划航线，但必须为氧气泄漏、设备故障、洋流突变分别准备应急预案。

#### 6.7.2 LLM 调用包装器

所有 LLM 调用必须通过统一的调用包装器，不允许直接裸调 LlamaIndex 的 LLM 接口。

```
LLM 调用包装器职责：
├── 重试机制：指数退避，最多重试 2 次（2s → 4s）
├── 超时控制：单次调用超时 60 秒
├── JSON 修复：返回非法 JSON 时自动尝试提取和修复
├── 空结果处理：返回空内容时视为"未发现问题"
├── 限流处理：429 响应时排队等待后重试
├── 异常捕获：所有异常不向上抛出，降级为安全默认值
└── 调用日志：记录每次调用的输入 token、输出 token、耗时、状态
```

#### 6.7.3 各场景容错策略

| 场景 | 检测方式 | 容错策略 | 对用户的影响 |
|------|---------|---------|-------------|
| LLM 调用超时（>60s） | 定时器 | 重试 2 次，仍超时标记该维度为"审核超时" | 该维度缺失，报告中标注 |
| LLM 返回非法 JSON | Pydantic 解析失败 | 正则提取 JSON 片段 → 重新解析 → 仍失败则丢弃 | 该 issue 丢失，日志记录 |
| LLM 返回空内容 | 内容长度 == 0 | 视为"未发现问题"，正常继续 | 无直接影响 |
| API 限流（429） | HTTP 状态码 | 指数退避等待（最多 30s），然后重试 | 审核时间变长 |
| 单个 Agent 整体失败 | 未捕获异常 | 该维度标记为"审核失败"，不阻塞其他 Agent | 该维度缺失 |
| 所有 Agent 失败 | 失败数 == Agent 总数 | 终止流程，保存检查点，提示用户重试 | 审核中断 |
| JSON 字段缺失 | Pydantic 校验 | 用默认值填充（score=0, severity=中） | 数据不完整 |

#### 6.7.4 JSON 修复策略

LLM 最常见的输出问题是 JSON 格式不合法。修复流程：

```
LLM 返回原始文本
    │
    ▼
直接 JSON 解析
    ├── 成功 → 返回
    └── 失败 ↓
    │
    ▼
正则提取最外层 {} 内容
    ├── 成功 → JSON 解析
    │   ├── 成功 → 返回
    │   └── 失败 ↓
    └── 失败 ↓
    │
    ▼
尝试修复常见问题：
  - 去除尾部逗号
  - 修复未转义的引号
  - 补全缺失的括号
    ├── 成功 → 返回
    └── 失败 → 返回 None，记录日志
```

#### 6.7.5 调用日志格式

```python
@dataclass
class LLMCallLog:
    """LLM 调用日志"""
    call_id: str                      # 调用唯一 ID
    agent_name: str                   # 调用者

```

### 6.8 评分体系设计

#### 6.8.1 设计理念

评分体系必须是**可计算、可解释、可配置**的。不能是 LLM 随便给一个数字。

- **可计算**：有明确的公式，同样的 issue 列表一定算出同样的分数
- **可解释**：每个扣分都能追溯到具体问题
- **可配置**：权重和扣分规则可以在 system_config.yaml 中调整

#### 6.8.2 评分公式

```
维度分数计算：
  维度基础分 = 100
  维度扣分 = Σ(每个 issue 的扣分)
  维度分数 = max(0, 维度基础分 - 维度扣分)

单个 issue 扣分规则：
  高严重度：扣 15 分
  中严重度：扣 5 分
  低严重度：扣 1 分

综合评分计算：
  综合评分 = Σ(维度分数 × 维度权重)

默认维度权重（可在 system_config.yaml 中配置）：
  完整性：0.35
  合规性：0.30
  报价：  0.20
  风险：  0.15

风险等级映射：
  综合评分 ≥ 85 → 低风险
  综合评分 60~84 → 中风险
  综合评分 < 60 → 高风险
```

#### 6.8.3 评分置信度修正

当 issue 的 confidence < 0.8 时，扣分打折：

```
实际扣分 = 基础扣分 × confidence 系数

confidence ≥ 0.9：不打折，扣 100%
confidence 0.8~0.9：打 8 折
confidence 0.7~0.8：打 6 折
confidence < 0.7：打 3 折（低置信度，大幅降低影响）
```

这确保了"有把握的问题影响大，没把握的问题影响小"。

#### 6.8.4 system_config.yaml 中的评分配置

```yaml
# ============ 评分配置 ============
scoring:
  base_score: 100
  
  severity_deduction:
    高: 15
    中: 5
    低: 1
  
  dimension_weights:
    完整性: 0.35
    合规性: 0.30
    报价: 0.20
    风险: 0.15
  
  confidence_factor:
    ">=0.9": 1.0
    "0.8-0.9": 0.8
    "0.7-0.8": 0.6
    "<0.7": 0.3
  
  risk_thresholds:
    低风险: 85
    中风险: 60
    # < 60 为高风险
```

### 6.9 错误处理与降级策略

#### 6.9.1 设计理念

系统必须为**每一种可预见的失败**准备降级方案。核心原则：**宁可降级输出，不可直接崩溃。**

类比深海潜水器：一个系统故障时，备份系统自动接管。没有备份？至少要有逃生通道。

#### 6.9.2 错误分类与处理

| 阶段 | 错误类型 | 检测方式 | 降级策略 | 用户感知 |
|------|---------|---------|---------|---------|
| 文件上传 | 文件损坏 | python-docx 异常 | 拒绝处理，提示"文件格式错误" | 错误提示 |
| 文件上传 | 文件过大（>100MB） | 文件大小检查 | 拒绝处理，提示大小限制 | 错误提示 |
| 文件上传 | 文件加密/密码保护 | 读取异常 | 提示用户解密后重新上传 | 错误提示 |
| 文档解析 | 章节无内容 | 内容长度 == 0 | 跳过该章节，报告中标注"空章节" | 无感知 |
| 文档解析 | 表格解析失败 | python-docx 异常 | 降级为纯文本，标记"表格可能不完整" | 低感知 |
| 文档解析 | 标题层级混乱 | 层级异常检测 | 按 flat 列表处理，不组装层级 | 低感知 |
| 行业分析 | Analyzer 失败 | LLM 调用异常 | 使用默认配方（4 个通用维度） | 低感知 |
| 章节摘要 | 摘要生成失败 | LLM 调用异常 | 跳过摘要，直接用原文（可能超长） | 低感知 |
| 粗筛扫描 | 粗筛失败 | LLM 调用异常 | 降级为全量精审（所有 chunk 都审） | 审核变慢 |
| 精细审核 | 单 chunk 审核失败 | LLM 调用异常 | 跳过该 chunk，标记"该区域未审核" | 部分缺失 |
| 精细审核 | 整个维度失败 | 所有 chunk 失败 | 该维度标记"审核失败"，不阻塞其他维度 | 维度缺失 |
| 结果校验 | 向量索引构建失败 | 异常捕获 | 跳过回查，所有低置信度 issue 标记"待人工确认" | 低感知 |
| 结果校验 | 回查验证失败 | LLM 调用异常 | 直接通过，不删除 issue | 低感知 |
| 报告生成 | 报告生成失败 | LLM 调用异常 | 使用模板填充，不依赖 LLM 生成文字 | 报告生硬 |
| Word 标注 | 标注失败 | python-docx 异常 | 跳过标注，只输出报告和 JSON | 低感知 |
| 检查点保存 | 磁盘空间不足 | IOError | 保留最近 3 个检查点，清理更早的 | 低感知 |
| 检查点恢复 | 检查点文件损坏 | JSON 解析失败 | 从更早的检查点恢复，或从头开始 | 审核重来 |

#### 6.9.3 全局降级等级

```
降级等级（从轻到重）：

Level 1 - 静默降级
  影响：输出质量略降，用户无感知
  示例：表格解析失败降级为纯文本
  处理：记录日志，继续执行

Level 2 - 标记降级
  影响：输出中明确标注降级部分
  示例：某 chunk 审核失败，报告中标注"该区域未审核"
  处理：记录日志，继续执行，报告中体现

Level 3 - 维度降级
  影响：整个审核维度缺失
  示例：合规性审核 Agent 整体失败
  处理：记录日志，其他维度继续，报告中标注该维度缺失

Level 4 - 流程中断
  影响：审核无法继续
  示例：所有 Agent 失败、文档解析完全失败
  处理：保存检查点，提示用户错误原因，建议重试
```

#### 6.9.4 错误处理统一接口

```
所有模块的错误处理遵循统一模式：

try:
    result = 执行操作
    return result
except 预期异常:
    执行降级策略
    记录日志
    return 降级默认值
except 未预期异常:
    记录完整堆栈
    执行最保守的降级策略
    return 安全默认值
    # 绝不向上抛出异常
```

## 八、LlamaIndex 使用设计

### 8.1 文档加载

```python
from llama_index.readers.file import DocxReader

reader = DocxReader()
documents = reader.load_data("标书.docx")
```

### 8.2 文本切分

```python
from llama_index.core.node_parser import SentenceSplitter

splitter = SentenceSplitter(
    chunk_size=8000,
    chunk_overlap=500
)
nodes = splitter.get_nodes_from_documents(documents)
```

### 8.3 LLM 调用

```python
from llama_index.llms.openai_like import OpenAILike

qwen_max = OpenAILike(
    model="qwen-max",
    api_base="https://dashscope.aliyuncs.com/compatible-mode/v1",
    api_key="your_key"
)
```

### 8.4 结构化输出

```python
from pydantic import BaseModel
from llama_index.core.program import LLMTextCompletionProgram

class ReviewResultSchema(BaseModel):
    issues: list[dict]
    summary: str
    score: float
    risk_level: str

program = LLMTextCompletionProgram.from_defaults(
    output_cls=ReviewResultSchema,
    prompt_template_str="{prompt}",
    llm=qwen_max
)
```

### 8.5 向量索引（用于回查验证）

```python
from llama_index.core import VectorStoreIndex

index = VectorStoreIndex(nodes)
query_engine = index.as_query_engine(similarity_top_k=3)
```

## 十二、扩展设计

### 12.1 新增业务场景

```
新增一个业务场景需要做的事：

1. 创建 scenes/新场景/ 目录
2. 实现 Scene 接口（约 200-300 行）
3. 编写 Skill 模板（纯文本配置）
4. 注册到平台

平台层代码零改动。复用率 70-80%。
```

### 12.2 新增文档格式

```
新增文档格式支持：

1. 在 LlamaIndex 中找到对应的 DocumentLoader
2. 封装到 infrastructure/document_loader.py
3. 注册到加载器工厂

业务层代码零改动。
```

### 12.3 新增行业支持

```
新增行业支持：

1. 在 prompts/skills/ 下新建行业目录
2. 编写行业专用 Skill 模板
3. 在 Analyzer 的行业识别规则中注册

代码逻辑零改动，只加配置。
```

### 12.4 新增 LLM 模型

```
新增 LLM 模型支持：

1. 在 LlamaIndex 中找到对应的 LLM Adapter
2. 封装到 infrastructure/llm_factory.py
3. 在 system_config.yaml 中指定哪些环节用哪个模型

业务逻辑零改动。
```

## 十四、v1.0 → v2.1 改进总结

| 维度 | v1.0 | v2.1 | 改进来源 |
|------|------|------|---------|
| 全局配置 | 无 | system_config.yaml | CLAUDE.md |
| Agent 协作 | 仅并行，无协作 | 消息总线 + 共享上下文 | Agent Teams |
| 长文档处理 | 摘要→审核（丢细节） | 两级审核（粗筛+精审） | Context Window 管理 |
| 校验机制 | 所有 Agent 完成后统一 LLM 校验 | Hooks 实时规则校验 + 按需 LLM | Hooks |
| 成本控制 | 全量 LLM 调用 | 规则优先，LLM 兜底 | Hooks |
| Prompt 管理 | 散落各处 | Skill 模板库 | Skills |
| 流程控制 | 线性执行 | 检查点 + 回滚 + 断点续审 | Checkpoints |
| 上下文管理 | 无 | Agent 独立上下文 + 智能压缩 | Sub Agents + /compact |
| 人工干预 | 有 | 有 + 干预即检查点 | Checkpoints |
| 输出统计 | 无 | Hook 执行统计 | Hooks |
| LLM 容错 | 无 | 统一调用包装器 + 重试 + JSON 修复 | v2.1 新增 |
| 评分体系 | 无定义 | 公式化评分 + 置信度修正 + 可配置权重 | v2.1 新增 |
| 错误处理 | 无 | 分级降级策略 + 全场景覆盖 | v2.1 新增 |

## 十五、一句话总结

```
用 LlamaIndex 处理文档和调用模型（基础设施）
用 system_config.yaml 统一全局规范（CLAUDE.md）
用 Agent Teams 编排多 Agent 协作（去中心化）
用 Hooks 做零成本规则校验（事件驱动）
用 Skills 管理可复用的专家知识（Prompt 模板库）
用 Checkpoints 保障流程可回滚（容错机制）
用两级审核策略解决长文档问题（Context 管理）
用人工干预保障审核质量（人机协同）
用 LLM 调用包装器保障稳定性（容错策略）
用公式化评分保障结果可解释（评分体系）
用分级降级策略保障系统不崩溃（错误处理）

先跑通标书审核，再扩展其他场景。
```
